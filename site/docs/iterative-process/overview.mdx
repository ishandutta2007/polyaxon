Most Polyaxon work happens in loops: change code, change params, run again, inspect the result. The right loop depends on where your code lives and how much of the run you want to drive from the CLI, a notebook, or Python.

Use [local code uploads](/docs/iterative-process/iterate-with-cli-local-code/) when you are editing files on your machine and want Polyaxon to package the current folder for each run.

Use [git-based runs](/docs/iterative-process/iterate-with-cli-git-code/) when the code should come from a repository. This is the better default for shared projects and reproducible runs.

Use [inline scripts](/docs/iterative-process/iterate-with-inline-scripts/) for small commands, debugging, or generated code that does not need its own repository.

Use [notebook iteration](/docs/iterative-process/iterate-in-notebooks/) when you want to run experiments from a notebook session and keep the resulting runs tracked in Polyaxon.

Use [programmatic iteration](/docs/iterative-process/iterate-programmatically/) when your loop lives in Python and you want to create runs through the client.

Use [presets](/docs/iterative-process/iterate-with-presets/) when several runs share the same queues, resources, connections, or environment settings.

Use [passing params](/docs/iterative-process/passing-params/with-args/) when you want to vary inputs without editing the component or operation definition.
